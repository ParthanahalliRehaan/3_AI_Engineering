# Staging & Deployment Workflow — Explained Clearly

The prof is teaching **staging** — a critical real-world concept. Here's everything broken down:

---

## The Problem He Starts With

The app is live (real users on it), but he needs to make a **big code change** — switching from SQLite to PostgreSQL.

The question is: **how do you safely update a live app?**

---

## The Wrong Approach ❌

```
Fix code → push to main → users get untested code
```

This is bad because:
- The new code hasn't been tested on real server infrastructure
- If something breaks, **real users are affected immediately**

---

## The Right Approach: Staging ✅

**Staging** = a private copy of your app that mimics production exactly, but only the dev team can access it.

```
Dev Machine → staging branch → staging server (private, test here)
                                       ↓ (only when happy)
                              main branch → production server (public)
```

Think of it like a **dress rehearsal** before the real show.

---

## The Full Workflow He Uses

```
1. Write new code
2. Push to a GitHub branch called "staging" (NOT main)
3. Set up a staging server on Render (points to staging branch)
4. Test everything works in real-world conditions
5. Fix issues, repeat testing as needed
6. When confident → merge staging into main
7. Render auto-redeploys production ✅
```

---

## Step-by-Step Git Commands

### Step 1 — Save your current code
```bash
git add .
git commit -m "switch to postgres"
```

### Step 2 — Create and switch to a new branch called `staging`
```bash
git checkout -b staging
```
The `-b` flag means **create this branch and switch to it**.

### Step 3 — Push the staging branch to GitHub
```bash
git push origin staging
```

Now on GitHub you'll see two branches: `main` and `staging`.

---

## Setting Up PostgreSQL on Render

### Why PostgreSQL instead of SQLite?
Already covered in the last lesson — concurrent writes, no file system dependency, works with multiple servers.

### Steps on Render.com:
1. Go to your project → **New Service → PostgreSQL**
2. Name it (e.g. `DCDB`)
3. Choose same region as your app (important!)
4. Select **free tier** → Create Database
5. Copy the **Internal Database URL** (works because app & DB are in same region)

---

## Connecting the Database to the App

In the app's code, the database connection looks like this:

```js
// database config file
const db = new Pool({
  connectionString: process.env.DATABASE_URL  // reads from environment variable
});
```

`process.env.DATABASE_URL` means — *read the database address from an environment variable*, not hardcoded. So on Render, you set:

```
Key:   DATABASE_URL
Value: (paste the internal postgres URL from Render)
```

This keeps secrets out of your code.

---

## Setting Up the Staging Web Service on Render

1. New Service → **Web Service**
2. Connect same GitHub repo
3. Name it `dc-staging`
4. **Change branch from `main` → `staging`** ← key step
5. Add environment variable: `DATABASE_URL` = your Postgres internal URL
6. Deploy

Now you have:

| Service | Branch | Who can access |
|---|---|---|
| `dc-staging.onrender.com` | `staging` | Just you/dev team |
| `dreamcatcher.onrender.com` | `main` | Everyone (public) |

---

## Testing on Staging

Once deployed, you test the staging URL thoroughly:
- Add data, delete data, check everything works
- If something breaks → fix it on the `staging` branch, push again
- Render auto-redeploys staging
- Keep testing until confident

---

## What Comes Next (Next Lesson)

Once staging is fully tested and working, you **merge into main**:

```bash
git checkout main
git merge staging
git push origin main
```

Render detects the push to `main` → auto-redeploys production → **all users get the new code safely**. ✅

---

## The Big Picture Takeaway

```
Never push untested code directly to production.

staging branch = safe sandbox
main branch    = what real users see
```

This is standard practice at every professional software company — now you know how it works from scratch.
