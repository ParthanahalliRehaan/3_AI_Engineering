# Logs, Rollbacks & Monitoring — Explained Clearly

This lesson is about **what to do when things go wrong** in production. Four tools are covered:

---

## 1. Logs

Just like `console.log` in the browser or Node, but on the server.

```js
// your server code
console.log("User added a dream:", dreamData)  // this shows up in Render's logs
console.error("DB connection failed!")          // this too
```

**Where to find them:** Render dashboard → your service → Logs tab

### ⚠️ Important limitation:
Logs are **not permanent**. They get wiped when:
- App redeploys
- App restarts
- App spins down (free tier goes to sleep after inactivity)

So logs are great for **live debugging**, but not for long-term monitoring.

---

## 2. Rollbacks

If you deploy new code and it breaks everything, you can **instantly go back** to the previous working version.

```
Deploy v3 (broken) ← you're here, things are broken
Deploy v2 (working) ← click rollback, go back here instantly
Deploy v1
```

On Render: Dashboard → your service → **Events tab** → click Rollback on any previous deploy.

One click, Render does the rest. No git commands needed.

---

## 3. Passive Notifications (Email / Slack)

You can't fix a problem you don't know about. Render can **automatically alert you** when something goes wrong.

Common events to get notified about:
- Deploy failed ← most important
- Service went down
- Deploy succeeded

**Where to set up:** Render dashboard → Notifications → choose Email or Slack

You can also set notifications **per service** — go into a specific service → Settings → scroll down → notification settings.

---

## 4. Webhooks (Advanced — Paid Tier)

This is the most powerful option. A webhook lets you **trigger your own code** when something happens on Render.

### How it works:

```
Something happens on Render (e.g. deploy fails)
        ↓
Render fires a "deploy.failed" event
        ↓
Webhook sends JSON to YOUR server
        ↓
Your server does whatever you want with it
```

### Example JSON Render might send:
```json
{
  "event": "deploy.failed",
  "service": "dreamcatcher",
  "timestamp": "2025-05-30T10:00:00Z",
  "details": "Build failed at step: npm install"
}
```

### What your server could do with that:

```js
app.post('/webhook', (req, res) => {
  const event = req.body

  if (event.type === 'deploy.failed') {
    updateStatusPage('🔴 Service is currently down')  // tell users
    notifyEngineer(event)                              // wake up a dev
  }

  res.sendStatus(200)
})
```

Render has ~50 different events you can hook into — deploys, database events, scaling events, etc.

---

## 5. Incident Management Tools (PagerDuty / Opsgenie)

These go one step beyond Slack notifications. Used by professional engineering teams.

| Tool | What it does |
|---|---|
| **PagerDuty** | Escalating alerts, on-call schedules, wakes engineers up at 3am if needed |
| **Opsgenie** | Similar — alert routing, on-call management |

```
Deploy fails
    ↓
Webhook fires
    ↓
PagerDuty gets notified
    ↓
On-call engineer gets a phone call at 3am 📱
    ↓
Engineer fixes it before most users notice
```

Overkill for solo projects, but standard practice at any serious software company.

---

## The Full Picture

```
Something breaks in production
        ↓
Logs     → see what error happened (short-term)
Rollback → instantly revert to last working version
Notifications → you find out immediately (email/Slack)
Webhooks → automate your response (status page, alerts)
PagerDuty → make sure the right person gets woken up
```

---

## Takeaway for Your Projects (Motrix / RAG app)

Even on free tier, at minimum you should:
1. **Set up email notifications** for deploy failures — free and takes 2 minutes
2. **Know where the logs are** so you can debug quickly
3. **Know rollback exists** — saves you when a bad deploy slips through staging
