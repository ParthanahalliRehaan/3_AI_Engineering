# Health Endpoints — Deeply Explained

---

## What is a Health Endpoint?

It's a special URL in your app — like `/health` — that answers one question:

> **"Is everything working right now?"**

It's not for users. It's for **tools and developers** to check the app's status programmatically.

---

## Who Uses It?

```
Monitoring tools (e.g. UptimeRobot)  → hits /health every 60s automatically
Load balancers                        → decides whether to route traffic to this server
Render/AWS/cloud platforms            → decides whether to restart the service
You (via cURL)                        → manually check if app + DB are alive
```

---

## The Full Code — With Explanation of Every Line

```js
const { Pool } = require('pg')  // import postgres connection pool

// pool lets you make DB queries
const pool = new Pool({
  connectionString: process.env.DATABASE_URL
})

// Health endpoint
app.get('/health', async (req, res) => {
  
  try {
    // "SELECT 1" is the simplest possible SQL query
    // It doesn't read any real data — it just pokes the database
    // If this line runs without error → database is alive ✅
    // If this throws an error → database is dead ❌
    await pool.query('SELECT 1')

    // If we reach here, everything is working
    res.json({
      status: 'ok',
      database: 'connected',
      uptime: process.uptime()  // how long the server has been running (in seconds)
    })

  } catch (error) {
    
    // Database couldn't be reached
    // 503 = "Service Unavailable" — server is UP but a dependency (DB) is DOWN
    res.status(503).json({
      status: 'error',
      database: 'disconnected',
      message: error.message,       // exact error e.g. "Connection refused"
      uptime: process.uptime()      // server itself is still running, so include this
    })
  }
})
```

---

## Why `SELECT 1`?

It's the most minimal valid SQL query. It does nothing except make a round-trip to the database.

```sql
SELECT 1
-- returns: 1
-- that's it. No tables, no data, just "hey DB, are you there?"
```

Works on **every** SQL database — PostgreSQL, MySQL, SQLite, etc. Very versatile.

---

## What `process.uptime()` Returns

```js
process.uptime()  // → 628.878 (seconds the Node server has been running)
```

This is useful because even if the DB is down, you can see:
- Server has been up for 628 seconds → server itself is healthy
- Only the database connection is the problem

---

## HTTP Status Codes Used

| Code | Meaning | When used here |
|---|---|---|
| `200` | OK | DB connected, everything fine |
| `503` | Service Unavailable | Server up, but DB unreachable |

Note: `503` doesn't mean your server crashed — it means **a dependency your server needs is unavailable**.

---

## What the Responses Look Like

### ✅ Healthy response (DB connected):
```json
{
  "status": "ok",
  "database": "connected",
  "uptime": 15.5
}
```

### ❌ Unhealthy response (DB disconnected):
```json
{
  "status": "error",
  "database": "disconnected",
  "message": "connect ECONNREFUSED 127.0.0.1:5432",
  "uptime": 628.878
}
```

Notice: uptime is included in **both** responses — because the server itself is still running even when the DB is down.

## Testing With cURL

`cURL` is a command line tool to make HTTP requests — like Postman but in the terminal.

### Locally (in development):
```bash
curl http://localhost:3001/health
```

### On production server:
```bash
curl https://your-app-name.onrender.com/health
```

### With prettier output (optional):
```bash
curl https://your-app-name.onrender.com/health | json_pp
# json_pp = "JSON pretty print" — formats the output nicely
```

## Health Endpoint Naming Conventions

| URL | Style | Used by |
|---|---|---|
| `/health` | Common, balanced | Most apps |
| `/healthz` | Minimalist with status | Google, Kubernetes |
| `/status` | Rich with lots of info | Large systems |
| `/ping` | Bare minimum, just "alive?" | Simple uptime checks |

They all do essentially the same job — pick one and be consistent.

---

## Connecting This to the Bigger Picture

```
Health endpoint exists
        ↓
Monitoring tool (e.g. UptimeRobot) hits it every 60 seconds
        ↓
Gets 503 (DB disconnected)
        ↓
Triggers webhook on Render
        ↓
PagerDuty wakes up the on-call engineer
        ↓
Engineer sees "database: disconnected, uptime: 4820s"
        ↓
Knows immediately: server is fine, DB needs fixing
```

Without the health endpoint, you'd just get a vague "app is down" alert with no clue where to start debugging.

## For Your Projects (Motrix / RAG App)

You should absolutely add this to both. A minimal version for Motrix's auth system:

```js
app.get('/health', async (req, res) => {
  try {
    await pool.query('SELECT 1')  // or your Supabase equivalent
    res.json({
      status: 'ok',
      database: 'connected',
      uptime: process.uptime(),
      environment: process.env.NODE_ENV  // handy to know if you're on staging or prod
    })
  } catch (err) {
    res.status(503).json({
      status: 'error',
      database: 'disconnected',
      message: err.message,
      uptime: process.uptime()
    })
  }
})
```

One endpoint, but it instantly tells you whether your auth system's database dependency is alive — incredibly useful when debugging production issues.
