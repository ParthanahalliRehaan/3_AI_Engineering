`NODE_ENV` can live at:

**.env file**
```
NODE_ENV=production
```
Read via `process.env.NODE_ENV` in Node.js (usually needs dotenv package to load it)

**package.json scripts**
```json
"scripts": {
  "start": "NODE_ENV=production node server.js",
  "dev": "NODE_ENV=development node server.js"
}
```
Here it's set inline when you run the script — no extra file needed.

**vite.config.js**
Vite has its own system — it uses `import.meta.env.MODE` instead of `process.env.NODE_ENV`, and you create `.env.development` or `.env.production` files to go alongside it.

**The key point** is that `NODE_ENV` (and env variables generally) can be set from **many different places** depending on your stack, and they all feed into the same idea — giving your code context about where it's running. The important thing to understand is:

- **Node/Express apps** → typically `.env` or `package.json` scripts
- **Vite/frontend tools** → their own config system
- **Hosting platforms** (like Railway, Vercel, Heroku) → you set them in a dashboard, no file needed at all
