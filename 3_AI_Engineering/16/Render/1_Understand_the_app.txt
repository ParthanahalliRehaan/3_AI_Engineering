# Your DreamCatcher App — Explained Like You're You

You already know `index.html`, `script.js`, `style.css`, and basic Express. Perfect. I'll explain everything **using only what you already know** as the starting point.

## First: What Even Is This Project?

Your old projects looked like this:
```
my-project/
  index.html
  style.css
  script.js
```
The browser opens `index.html`, reads `style.css` for looks, runs `script.js` for behaviour. Done.

DreamCatcher is bigger. It has **two separate worlds**:

- **The browser world** — `index.html`, `styles.css`, `app.js` (your front end, stuff the user sees)
- **The server world** — `server.js` and all the other `.js` files (stuff running on your computer/server, invisible to the user)

The browser world and the server world **talk to each other through URLs** — just like when your `script.js` used to do `fetch('https://some-api.com')`. Except now *you* built the API.

## What Is a Database? (Like You're 12)

You know how Excel has rows and columns? A database is the same idea, but:
- It lives in a file on your computer (`dreams.db` in this project)
- Your JavaScript can talk to it: "save this", "get all rows", "delete row #5"
- It survives when your server restarts — unlike a regular JavaScript variable which disappears

Your project uses **SQLite** — the most beginner-friendly database. It's literally just one file: `dreams.db`. No special server needed, no passwords. Just a file that behaves like a super-powered spreadsheet.

The table in your database looks like this:

| id | dream_text | interpretation | created_at |
|----|-----------|---------------|------------|
| 1 | "I was flying..." | "Flying often means..." | 2024-01-01 |
| 2 | "A snake chased me..." | "Snakes can symbolize..." | 2024-01-02 |

## What Is a "DB" File? (`config/database.js`)

```js
// This is the file that knows HOW to open the database
export async function getDatabase() {
  if (!db) {
    db = await open({
      filename: dbPath,   // "which file?"  →  dreams.db
      driver: sqlite3.Database  // "what kind of database?"  →  SQLite
    });
  }
  return db;  // hand back the open database, ready to use
}
```

Think of this file like a **key-maker**. Every other file that needs to touch the database calls `getDatabase()` and gets back a key to use it.

You'll never call this file directly — other files call it.

---

## What Is `database-init.js`?

```js
export async function initDatabase() {
  const db = await getDatabase();
  await db.exec(`
    CREATE TABLE IF NOT EXISTS dreams (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      dream_text TEXT NOT NULL,
      interpretation TEXT,
      created_at DATETIME DEFAULT CURRENT_TIMESTAMP
    )
  `);
}
```

This runs **once when your server starts**. It says: *"Hey database, if you don't have a `dreams` table yet, create one."*

The stuff inside the backticks is **SQL** — the language databases speak. You don't need to fully learn it today, but here's what those words mean in plain English:

- `CREATE TABLE IF NOT EXISTS dreams` → Make a spreadsheet called "dreams", but only if one doesn't already exist
- `id INTEGER PRIMARY KEY AUTOINCREMENT` → Every row gets a unique number, counting up automatically (1, 2, 3...)
- `dream_text TEXT NOT NULL` → A column for the dream text — can't be empty
- `interpretation TEXT` → A column for the AI's answer — *can* be empty
- `created_at DATETIME DEFAULT CURRENT_TIMESTAMP` → Automatically stamps the date/time when saved

## What Is `express.Router()`? (The Thing You Don't Know Yet)

You know basic Express:
```js
// What you know
const app = express();
app.get('/dreams', (req, res) => { ... });
app.post('/dreams', (req, res) => { ... });
```

That works fine! But when you have 20 different URLs, putting them all in `server.js` gets messy. **A Router is just a way to move those URL handlers into their own file** and plug them in.

In `routes/dreams.js`:
```js
const router = express.Router(); // It's like a mini express app

router.get('/', ...)    // handles GET /api/dreams
router.post('/', ...)   // handles POST /api/dreams
router.delete('/:id', ...) // handles DELETE /api/dreams/5

export default router; // Send it out for server.js to use
```

In `server.js`:
```js
import dreamsRouter from './routes/dreams.js'; // bring it in
app.use('/api/dreams', dreamsRouter); // plug it in at this URL prefix
```

That's it. `express.Router()` = "let me write my URL handlers in a separate file." Nothing more magical than that.

## How the Whole Thing Works Together (The Flow)

Here's what happens when a user types a dream and clicks "Get Interpretation":

```
1. USER clicks the button in the browser
        ↓
2. app.js (public/app.js) does:
   fetch('/api/dreams', { method: 'POST', body: { dream_text: "I was flying..." } })
        ↓
3. server.js receives the request, sees it's for /api/dreams
   → passes it to routes/dreams.js
        ↓
4. routes/dreams.js runs the POST handler:
   a. validateText() checks: is it empty? too short? too long?
   b. calls getDreamInterpretation() from utils/ai-openai.js
        ↓
5. utils/ai-openai.js sends the dream to OpenAI's servers
   → OpenAI responds with an interpretation
        ↓
6. Back in routes/dreams.js:
   → saves dream_text + interpretation to the database (dreams.db)
   → sends the saved dream back as a response
        ↓
7. app.js in the browser receives the response
   → shows it on the page
```

## What Is an ORM? (The Word the Scrimba Prof Threw At You)

An ORM is a fancier way to talk to a database — instead of writing `SELECT * FROM dreams`, you write JavaScript like `Dream.findAll()`. 

**Your project does NOT use an ORM.** It writes raw SQL directly. That's actually more beginner-friendly! You can see exactly what's happening:

```js
// This is what your project does (raw SQL — clear and simple):
const result = await db.all('SELECT * FROM dreams ORDER BY created_at DESC');

// An ORM would hide that and let you write:
const result = await Dream.findAll({ order: [['created_at', 'DESC']] });
```

Both do the same thing. Your project's way is easier to understand.

## The `.env` Thing — What's That API Key Doing in the Code?!

Look at `utils/ai-openai.js`:
```js
const openai = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY || 'sf-proj-nklnlhn...'  
  //                                    ↑ THIS IS A PROBLEM
});
```

**That hardcoded API key should never be in your code.** It's like leaving your password in a text file. Anyone who sees your code can steal your key and rack up charges on your account.

The right way:

1. Create a file called `.env` in your project root:
```
OPENAI_API_KEY=sk-your-real-key-here
```

2. Add `.env` to a file called `.gitignore`:
```
.env
node_modules
```

3. Now `process.env.OPENAI_API_KEY` in your code reads from that file — but the file never gets uploaded anywhere.

`.gitignore` tells Git: "pretend these files don't exist." Git is the thing that tracks your code history and lets you upload to GitHub. The `.env` file stays only on your computer.

## What Is `package.json`?

```json
{
  "name": "dream-catcher",
  "dependencies": {
    "express": "^4.18.2",
    "sqlite": "^5.1.1",
    "openai": "^4.52.0"
  }
}
```

This is the **shopping list**. When you run `npm install`, npm reads this file and downloads everything listed into a folder called `node_modules`.

`node_modules` is HUGE (thousands of files). You never upload it anywhere — it's in `.gitignore`. Anyone who gets your code just runs `npm install` and rebuilds it from the shopping list.

---

## What Is `"type": "module"` in `package.json`?

```json
"type": "module"
```

This one line is why the project uses `import`/`export` instead of the old `require()` style. It tells Node: "treat all my `.js` files as modern ES modules." 

If you've written `import` in your browser JS before, this makes Node work the same way.

---

## What Is `nodemon`?

```json
"scripts": {
  "dev": "nodemon server.js"
}
```

When you run `npm run dev`, it starts your server with **nodemon** watching. Every time you save a file, it automatically restarts the server. Without it, you'd have to manually stop and restart it every single time you changed your code. It's just a convenience tool for development.

---

## What Is `helmet`?

```js
import helmet from 'helmet'
// app.use(helmet());  ← it's commented out
```

Helmet adds security headers to every response your server sends. Headers are invisible instructions that tell the browser things like "don't let other websites load this page in an iframe." It's a one-liner that protects against common attacks. (It's commented out probably because it was causing issues with loading the fonts — easy fix but left for later.)

## The One Thing To Fix Right Now

**Remove that hardcoded API key from `ai-openai.js`.** Seriously. Even if you never publish this code anywhere, it's a bad habit to start.

1. Create a `.env` file next to `server.js`:
```
OPENAI_API_KEY=your-actual-key
```

2. Install dotenv: `npm install dotenv`

3. At the top of `server.js`, add:
```js
import 'dotenv/config';
```

4. In `ai-openai.js`, change to:
```js
const openai = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY
});
```

5. Create `.gitignore`:
```
.env
node_modules
dreams.db
```
