## 2. The Big Problem: Database Gets Wiped on Every Redeploy

After redeploying, **all his data disappeared**. Here's why:

### Render has an **Ephemeral File System**

> "Ephemeral" = temporary. It resets every time the app redeploys.

His database was **SQLite** ‚Äî which is just a file sitting in the project folder:

```
my-app/
‚îú‚îÄ‚îÄ index.html
‚îú‚îÄ‚îÄ server.js
‚îî‚îÄ‚îÄ database.sqlite   ‚Üê this is just a FILE. It gets wiped on redeploy.
```

Every redeploy = fresh code from GitHub = empty database again. Users lose all their data. Bad.

---

## 3. Two Solutions

### ‚úÖ Solution 1: Persistent Disk
A special storage area that **doesn't get wiped** on redeploy. Available on Render's paid plan.

```
GitHub code  ‚Üí  Render server (gets replaced on redeploy)
                     +
             Persistent Disk  ‚Üê database lives here, never wiped
```

### ‚úÖ Solution 2 (what he chose): Switch from SQLite ‚Üí PostgreSQL

---

## 4. Why SQLite is Bad for Production Web Apps

### Problem 1: Write Locking
SQLite locks the **entire database** when one user writes to it:

```
User A writes  ‚Üí  database LOCKED  ‚Üí  User B has to wait
                                    ‚Üí  User C has to wait
                                    ‚Üí  User D has to wait
```

With millions of users, this becomes a massive bottleneck.

### Problem 2: No Clustering
If your app gets huge, you run multiple servers. But SQLite is a local file, so each server gets its own copy:

```
Server 1 ‚Üí database.sqlite (has User A's data)
Server 2 ‚Üí database.sqlite (has User B's data)  ‚Üê different data! üíÄ
Server 3 ‚Üí database.sqlite (has User C's data)
```

This is a complete mess ‚Äî your data is fragmented across servers.

---

## 5. Why PostgreSQL Fixes Everything

### Concurrent writes ‚Äî no problem:
```
User A writes ‚îÄ‚îê
User B writes ‚îÄ‚î§‚Üí PostgreSQL handles all of them simultaneously ‚úÖ
User C writes ‚îÄ‚îò
```

### Works with multiple servers:
```
Server 1 ‚îÄ‚îê
Server 2 ‚îÄ‚î§‚Üí ONE shared PostgreSQL database ‚úÖ
Server 3 ‚îÄ‚îò
```

---

## 6. When IS SQLite actually good?

The prof defended SQLite ‚Äî it's great for:

| Use Case | Why SQLite Works |
|---|---|
| Mobile apps | One user at a time |
| Desktop apps | Single machine |
| IoT devices | Low/no concurrency |
| Simple blogs | Rare writes |

**Rule of thumb:** *One machine, one user at a time ‚Üí SQLite is perfect.*
